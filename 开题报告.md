# 文档理解与多模态检索问答系统

### ——基于 qwen3-vl-8b-instruct + LangChain 的多模态文档问答系统设计，以 GraphRAG 作为增强方向

---

## 一、任务定义与应用背景

### 1.1 任务定义

本项目面向课程题目 3，拟构建一个支持 PDF、扫描文档、PPT 截图等输入的**多模态文档问答系统**。整体设计采用“两层目标”推进：

**最低可交付版本（必做）**

1. 支持上传多页 PDF 或页面截图，并完成页面级解析与内容分块；
2. 支持识别文本、表格、图表等主要内容类型，并保留页码、图号等元数据；
3. 基于向量检索实现多模态 RAG 问答，回答中标注引用来源；
4. 完成纯文本 RAG 与多模态 RAG 的对比评测，满足课程题目 3 的最低验收要求。

**增强目标（进阶）**

1. 在基础多模态 RAG 之上引入文档知识图谱，增强跨页、多跳推理能力；
2. 进一步构建文本 GraphRAG 与多模态 GraphRAG，用于分析图结构带来的增益；
3. 尝试输出更细粒度的关系路径引用，例如“经由图号引用”或“经由同页/跨节关系”。

因此，本项目的核心研究问题被收敛为：**在先满足课程题目 3 基本要求的前提下，多模态信息能否显著提升文档问答效果；进一步引入图结构后，是否能够在多跳推理问题上提供额外收益？**

### 1.2 应用背景

随着数字化浪潮加速，企业、科研机构积累了海量图文混排文档（财报、学术论文、技术手册、法律合同等）。文档视觉问答（DocVQA）面临多重挑战：一是文档本身多模态（文本、图像、表格混排）；二是文档内容存在丰富的跨页逻辑关系（"第3节的图5引用了第1节的数据"）；三是传统向量检索仅靠语义相似度，无法捕获这种结构化的、需要多步推理的关系。

检索增强生成（RAG）通过引入外部知识库增强了模型的事实性；而知识图谱增强的GraphRAG进一步捕捉了实体间的结构化关系，使系统能够处理多跳推理。在此基础上融入多模态理解，是当前文档AI领域最前沿的研究方向。

具体应用场景包括：

- **学术科研**：追踪论文中"图5的结论来自第3节哪个实验"等跨章节逻辑链条
- **金融分析**：从财报中提取"图3的营收趋势与表4的分产品数据之间的关联"
- **法律合规**：定位"附件二中的图表与正文第5条款的引用关系"
- **企业知识管理**：跨文档、跨图表的多跳知识检索

## 二、相关工作调研

### 2.1 代表性数据集

**① DocVQA（Mathew et al., WACV 2021）**

DocVQA 是面向文档图像的视觉问答数据集，包含在 12,000+ 张文档图像上定义的 50,000 个问题。该任务超越了单纯的OCR，需要理解文档传递的各类信息，包括手写/打印文本、非文字元素（标记、勾选框、图表）、版面结构（表单、表格）以及字体样式等。DocVQA 是本项目评测的核心基准之一。

**② ChartQA（Masry et al., ACL Findings 2022）**

图表在数据分析中极为普遍，人们常提出涉及逻辑运算和视觉特征的复杂推理问题。ChartQA 提供了 9,600 道人工编写问题与 23,100 道基于图表摘要生成的问题，是目前规模最大的图表问答基准之一。本项目将使用 ChartQA 子集评测系统在图表理解上的能力。

### 2.2 代表性模型与方法

**③ qwen3-vl-8b-instruct（Alibaba Qwen Team, 2025）**

qwen3-vl-8b-instruct 在理解文档和图表方面具有显著优势，支持对发票、表单、表格等扫描件进行结构化输出，并且无需任务特定微调即可作为视觉代理（visual agent）使用。其视觉编码器为经过重新设计的 ViT 架构，引入了 2D-RoPE 与窗口注意力机制，原生支持可变输入分辨率。本项目通过阿里云百炼（DashScope）API 调用该模型，作为文档解析与多模态生成的双重引擎。

**④ ColPali（Faysse et al., 2024）**

现代检索系统主要依赖从文档页面提取的文本信息进行索引，这一过程往往繁琐且脆弱，难以高效利用视觉线索。ColPali 是一个经过训练的视觉语言模型，能够从文档页面图像中生成高质量的多向量嵌入；结合晚交互（late interaction）匹配机制，ColPali 性能大幅超越现代文档检索管道。本项目的视觉优先检索设计参考了其核心思想。

**⑤ MMDocRAG（2025）**

MMDocRAG 研究指出：视觉检索器更擅长检索图像内容，文本检索器擅长文本内容，混合检索器能提供更均衡的解决方案。这一结论直接支持了本项目采用"文本向量检索 + 视觉原图二次理解"混合架构的设计决策。

**⑥ Microsoft GraphRAG（Edge et al., arXiv 2404.16130, 2024）**

GraphRAG 由微软研究院提出，用以解决 LLM 在处理私有结构化数据时缺乏理解实体关系能力的问题，通过图结构建模、社区划分与查询增强机制来支持复杂检索和总结任务。该方法在生产级场景中体现了图增强检索对复杂问题的价值。本项目借鉴其“用图结构增强 RAG”的核心思想，但不完整复现其 community-based indexing、community report 和 global search 流程，而是面向课程作业构建轻量的文档图增强方案。

**⑦ Query-Driven Multimodal GraphRAG（ACL 2025 Findings）**

当前 RAG 和 GraphRAG 受限于静态知识库和低效的多模态数据整合；为此，研究者提出了查询驱动多模态 GraphRAG 框架，动态构建适配查询语义的局部知识图谱，通过：① 从查询语义推导图模式以引导知识提取；② 多路径检索策略精准定位核心知识；③ 按需补充缺失的多模态信息。本项目的图游走检索设计与该框架的多路径检索策略一脉相承。

**⑧ MMGraphRAG（arXiv 2025）**

MMGraphRAG 展示了将视觉信息与图结构结合用于文档理解的潜力，尤其适合处理跨页、多模态证据融合问题。该工作为本项目设置 GraphRAG 增强路线提供了参考，但本课程项目仍将以可交付、可验证的基础系统为优先目标。

### 2.3 关键技术横向对比

|维度|纯文本RAG|多模态RAG|文本GraphRAG|**多模态GraphRAG（本项目）**|
|---|---|---|---|---|
|文档解析|`get_text()` 纯文字|VLM结构化JSON|`get_text()` 纯文字|**VLM结构化JSON + 实体关系抽取**|
|检索机制|向量检索|向量检索|向量 + 图游走|**向量 + 图游走**|
|图表理解|❌|✅ VLM描述|❌|**✅ VLM描述 + 原图**|
|多跳推理|❌|❌|✅|**✅**|
|生成模型|Qwen-Turbo|qwen3-vl-8b-instruct|Qwen-Turbo|**qwen3-vl-8b-instruct**|

其中，**课程最低可交付版本以前两列为主**；后两列属于本项目希望进一步完成的增强实验，用于分析图结构的独立贡献。这里的 `GraphRAG` 指的是面向单文档问答的轻量图增强检索，而不是微软 GraphRAG 的完整工程复现。

## 三、系统方案设计

### 3.1 整体架构

为降低实现风险，本项目采用**分阶段架构**。第一阶段先完成满足课程题目要求的多模态 RAG 主链路；第二阶段在主链路稳定后，再加入知识图谱作为增强模块。当前增强部分聚焦于**面向具体问题的局部图检索（local-style search）**，不实现基于社区报告的全局主题总结。

text

```
【阶段一：最低可交付版本】
用户上传 PDF / 图像
    │
    ▼
[PyMuPDF] 渲染页面图片
    │
    ▼
[qwen3-vl-8b-instruct API] 提取 texts / tables / figures
    │
    ▼
构建带元数据的 LangChain Document
    │
    ▼
[DashScope text-embedding-v3] 向量化
    │
    ▼
[FAISS 本地向量库]
    │
    ▼
用户提问 → 检索相关块与原页面图片
    │
    ▼
[qwen3-vl-8b-instruct API] 生成带页码/图号引用的回答
    │
    ▼
[Gradio 界面] 展示

【阶段二：增强目标】
在阶段一基础上追加：
    │
    ├─ 抽取 entities / relations
    ├─ 构建 networkx 文档图
    ├─ 增加图游走扩展检索
    └─ 输出关系路径级引用
```

### 3.2 核心模块说明

**A. 必做模块**

**模块1：文档解析（parse_doc.py）**

使用 `PyMuPDF` 将 PDF 每页渲染为 PNG 图片。将页面图像传入 `qwen3-vl-8b-instruct` API，提取 `texts`、`tables`、`figures` 等结构化内容。第一阶段优先保证文本、表格、图表描述三类内容稳定输出；实体与关系抽取属于第二阶段增强项。

**模块2：向量化入库（build_vectorstore.py）**

使用 `DashScopeEmbeddings`（`text-embedding-v3`）将所有 Document 向量化，通过 `FAISS.from_documents()` 构建索引并持久化。每个向量条目携带页码、图号、类型等元数据，便于后续引用追踪。

**模块3：多模态 RAG 问答链（rag_chain.py）**

FAISS top-k 检索 → 读取对应页图像 → 组装多模态 Prompt → 调用 `qwen3-vl-8b-instruct` 生成带引用标注的回答。这一模块对应课程题目 3 中的核心能力，也是最低可交付版本的主链路。

**模块4：纯文本 RAG 基线（baseline_text_rag.py）**

使用 `page.get_text()` 提取纯文本，构建独立向量索引，再由纯文本模型生成回答。该模块用于满足题目要求中的“纯文本 RAG vs 多模态 RAG”对比。

**模块5：界面（app.py）**

基于 `Gradio` 构建 Web 界面，支持文件上传、索引构建、问题输入和结果展示。若时间紧张，界面可保持简单，但必须保证演示完整闭环。

**B. 增强模块**

**模块6：知识图谱构建（build_graph.py）**

用 `networkx` 初始化文档图，以 `node_id` 为节点标识添加内容节点及其属性。边分两类：

- 规则边：同页归属、图号引用、章节引用；
- 语义边：VLM 抽取的实体关系。

实现策略上优先保证规则边可用，语义边作为增强项加入；如果语义关系噪声较大，系统仍可退化为“规则图 + 向量检索”。本项目图谱为轻量文档图，不包含社区划分、社区报告生成和层级摘要等完整微软 GraphRAG 组件。

**模块7：文本 GraphRAG（text_graph_rag_chain.py）**

复用 `graph.pkl` 与 FAISS 双轨检索，仅传文本证据，用于分析图结构在无视觉输入条件下的效果。

**模块8：多模态 GraphRAG（multimodal_graph_rag_chain.py）**

在多模态 RAG 的基础上追加图游走扩展，将图结构证据、检索文本和页面原图一并传给视觉语言模型，探索其对多跳推理的帮助。该模块只面向具体问答场景下的局部证据扩展，不承担全局主题发现与社区级摘要任务。

**模块9：评测（evaluation.py）**

读取测试题集，先完成必做的两组对比；若增强模块落地，再扩展到四组系统与消融分析。

### 3.3 模型选择理由

|组件|选型|理由|
|---|---|---|
|视觉理解与文档解析|qwen3-vl-8b-instruct（DashScope API）|文档/图表理解领域顶级性能；支持结构化JSON输出；无需本地GPU|
|知识图谱|networkx（本地，纯Python）|零额外成本，轻量级，无需数据库服务；支持节点属性与有向边，满足文档图建模需求|
|文本嵌入|DashScope text-embedding-v3|与VLM同平台，延迟低，支持中英文，免费额度充足|
|向量数据库|FAISS（CPU版）|轻量级本地存储，`pip install faiss-cpu` 即用|
|纯文本对照LLM|Qwen-Turbo|价格低、调用方便，适合课程项目中的纯文本基线；但与 qwen3-vl-8b-instruct 不构成严格控制变量实验|
|文档渲染|PyMuPDF|纯CPU高速PDF转图片|

说明：纯文本组与多模态组使用的模型能力并不完全等价，因此本项目中的对比更偏向**工程比较与课程实验**，不将其表述为严格因果消融。

### 3.4 评测指标

|指标|描述|适用场景|
|---|---|---|
|**准确率（Accuracy）**|人工判断三分制（全对/部分对/错）|必做，40题全部|
|**引用准确率**|回答中引用的页码/图号是否正确|必做，含视觉元素题目|
|**ANLS**|平均归一化编辑相似度（DocVQA标准指标）|可选，用于精确答案题目|
|**多跳命中率**|系统是否正确追溯跨页引用链条|增强实验，多跳推理类|
|**图结构独立贡献**|文本 GraphRAG 与纯文本 RAG 的差异|增强实验|
|**视觉独立贡献**|多模态 RAG 与纯文本 RAG 的差异|必做对比，可重点分析图表/表格题|

## 四、初步数据集与实验设想

### 4.1 评测数据集方案

**数据集A：公开学术论文 PDF（3份）**

- 选取3篇含有丰富图表（≥5幅图、≥3个表格、≥2个跨节引用关系）的中英文学术论文PDF
- 推荐来源：arXiv.org 计算机视觉/NLP方向近期论文

**数据集B：ChartQA 子集**

- 从 ChartQA（9,600道人工题）中抽取10道图表推理题用于评测

**数据集C：DocVQA 子集**

- 从 DocVQA（50,000道题）验证集中抽取10道结构化文档题

**自建测试集（`test_questions.json`）**：针对数据集A的3份PDF，人工编写40道测试题：

|问题类型|数量|示例|
|---|---|---|
|纯文本理解|10题|"第3节的主要结论是什么？"|
|图表理解|10题|"图4中哪个类别的准确率最高？"|
|表格查询|10题|"表2中模型A在哪个指标上得分最低？"|
|**多跳推理**|**10题**|**"图3引用的数据来自第几节的哪个实验？"**|

### 4.2 实验设计

**实验一：课程要求对应的基础对比实验（必做）**

对比两组系统：

- 对照组 A：纯文本 RAG
- 对照组 B：多模态 RAG

重点观察：

- 图表题、表格题中，多模态证据是否优于纯文本证据；
- 回答能否稳定标注页码/图号引用；
- 系统是否已满足课程题目 3 的最低验收标准。

**实验二：图结构增强实验（选做）**

在基础两组实验跑通后，继续增加：

- 对照组 C：文本 GraphRAG
- 对照组 D：多模态 GraphRAG

重点观察：

- 图结构是否对多跳推理题有帮助；
- 在图结构加入后，引用链条是否更清晰；
- 若收益不明显，保留为探索性结果，不影响主线系统交付。

**实验三：引用标注与案例分析**

对所有回答检查引用标注格式是否正确，重点记录：

- 正确案例：能够定位页码、图号、表格；
- 错误案例：检索偏移、图表理解错误、关系链噪声；
- 典型失败模式：为最终汇报提供可视化和误差分析材料。

### 4.3 算力预算

|资源|需求|方案|
|---|---|---|
|本地计算|CPU-only|普通轻薄本（8GB RAM 即可）|

|**总计**|**无需GPU**|**预计在可接受的课程项目成本范围内**|

### 4.5 风险与回退方案

|风险点|影响|回退方案|
|---|---|---|
|VLM 结构化输出不稳定|页面解析结果不统一，影响后续索引|优先保证 `texts`、`tables`、`figures` 三类字段稳定，复杂字段延后|
|实体关系抽取噪声大|GraphRAG 效果不稳定，关系路径可能误导回答；同名实体、别名、缩写在跨页场景下也可能无法稳定对齐|先使用规则边（同页、图号引用、章节引用），语义边仅作为增强项；必要时接受“轻量文档图”而非严格知识库|
|四组系统同时实现时间不足|增强实验无法完整落地|优先保住“纯文本 RAG + 多模态 RAG + 基础评测 + 演示界面”|
|对照实验不够严格|答辩时可能被追问模型公平性|明确说明该实验偏工程比较，不将其表述为严格控制变量实验|
|评测样本不足或人工评分成本高|结果分析说服力下降|优先完成小规模但可复现的自建测试集，并补充典型案例分析|

## 五、预期创新点与贡献

1. **主线交付与增强探索分层明确**：先完成课程题目 3 所要求的多模态文档问答闭环，再在同一系统上逐步加入 GraphRAG，降低课程项目实施风险；
2. **文档图像直接驱动结构化抽取**：利用 VLM 从页面图像中提取文本、表格、图表描述，减少传统 OCR + 规则流水线的复杂度；
3. **图结构作为增强能力而非单点前提**：将知识图谱设计为增量模块，既保留多跳推理探索空间，也允许系统在图谱效果不足时稳定退回基础版本；
4. **面向课程汇报的可分析实验设计**：除了基础准确率对比，还会整理引用标注质量和典型失败案例，增强最终汇报的可解释性；
5. **低资源可落地**：全流程基于 API 调用与轻量本地组件，目标是在无本地 GPU 条件下完成可演示原型。
