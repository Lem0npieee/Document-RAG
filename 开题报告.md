# 期中开题报告

## 文档理解与多模态检索问答系统

### ——基于 Qwen2.5-VL + LangChain RAG 的智能文档助手

---

## 一、任务定义与应用背景

### 1.1 任务定义

本项目旨在构建一个**文档智能助手（Document Intelligent Assistant）**，支持用户上传 PDF、扫描文档、PPT 截图等多种文档形式，系统能够：

1. **自动解析文档结构**：识别并定位文本段落、表格、图表、公式等不同类型的内容区域；
2. **结构化分块入库**：将解析结果以带有来源元数据（页码、图号、类型）的向量块形式存入向量数据库；
3. **自然语言检索问答**：接收用户自然语言提问（如"图3表示什么趋势？""第2节主要结论是什么？"），系统检索相关证据并生成标注引用来源的回答；
4. **多模态证据融合**：回答同时依据检索到的文字描述与原始页面图像，实现文字+视觉的双重证据支撑。

核心研究问题是：**在面向文档的检索增强问答场景下，多模态RAG（Multimodal RAG）相比纯文本RAG能带来多大的效果提升？在哪些问题类型上差距最显著？**

### 1.2 应用背景

随着数字化浪潮加速，企业、科研机构积累了海量的图文混排文档（财报、学术论文、技术手册、法律合同等）。文档视觉问答（DocVQA）面临双重挑战：一是需要处理包含文本、图像、表格的冗长多模态文档，二是需要跨模态推理。[1](https://mmdocrag.github.io/MMDocRAG/)

检索增强生成（RAG）将外部知识库的丰富性与大型语言模型的生成能力相结合，为用户提供更准确的实时响应。然而，在信息爆炸时代，信息的呈现方式日益多模态化，仅靠传统文本知识库已无法满足需求，构建高效、准确的多模态RAG问答系统具有重要的理论与实践价值。[2](https://openreview.net/forum?id=fMaEbeJGpp)

具体应用场景包括：

- **学术科研**：快速定位论文中的图表结论，辅助文献综述
- **金融分析**：从财报PDF中提取关键数据与趋势图
- **法律合规**：从合同文档中检索关键条款与附件图表
- **企业知识管理**：统一检索跨格式的内部文档

## 二、相关工作调研

### 2.1 代表性数据集

**① DocVQA（Mathew et al., WACV 2021）**

DocVQA 是面向文档图像的视觉问答数据集，包含在 12,000+ 张文档图像上定义的 50,000 个问题。[3](https://arxiv.org/abs/2007.00398)该任务超越了单纯的OCR，需要理解文档传递的各类信息，包括手写/打印文本、非文字元素（标记、勾选框、分隔符、图表）、版面结构（表单、表格）以及字体样式等。[4](https://rrc.cvc.uab.es/?ch=17)DocVQA 是本项目评测的核心基准之一。

**② ChartQA（Masry et al., ACL Findings 2022）**

图表在数据分析中极为普遍，人们在探索图表时常提出涉及逻辑运算和视觉特征的复杂推理问题。[5](https://arxiv.org/abs/2203.10244)ChartQA 提供了 9,600 道人工编写问题与 23,100 道基于人工图表摘要生成的问题，是目前规模最大的图表问答基准之一。[5](https://arxiv.org/abs/2203.10244)本项目将使用 ChartQA 子集评测系统在图表理解上的能力。

### 2.2 代表性模型与方法

**③ Qwen2.5-VL（Alibaba Qwen Team, 2025）**

Qwen2.5-VL 是 Qwen 系列的最新成果，遵循开源理念，在多项基准上达到甚至超越了顶级闭源模型。[6](https://arxiv.org/pdf/2502.13923)Qwen2.5-VL 在理解文档和图表方面具有显著优势，并且无需任务特定微调即可作为视觉代理（visual agent）使用。[7](https://qwenlm.github.io/blog/qwen2.5-vl/)其架构创新主要体现在：视觉编码器为经过重大重新设计的 Vision Transformer（ViT）架构，引入了 2D-RoPE 与窗口注意力机制，原生支持可变输入分辨率并提升了计算效率。[8](https://debuggercafe.com/qwen2-5-vl/)此外，Qwen2.5-VL 支持对发票、表单、表格等扫描件进行结构化输出，在金融、商业等场景中极具实用价值。[9](https://ollama.com/library/qwen2.5vl)本项目将通过阿里云百炼（DashScope）API 调用该模型作为多模态理解的核心引擎。

**④ ColPali（Faysse et al., 2024）**

文档是通过文字、图形、版面、表格乃至字体传递信息的视觉丰富结构。然而，现代检索系统主要依赖从文档页面提取的文本信息进行索引，这一过程往往繁琐且脆弱，难以高效利用视觉线索，限制了其在RAG等实际文档检索应用中的能力。[10](https://arxiv.org/abs/2407.01449)ColPali 是一个经过训练的视觉语言模型，能够从文档页面图像中生成高质量的多向量嵌入；结合晚交互（late interaction）匹配机制，ColPali 的性能大幅超越现代文档检索管道，同时更简单、更快速、支持端到端训练。[10](https://arxiv.org/abs/2407.01449)ColPali 代表了视觉文档检索的前沿方向，本项目的检索设计参考了其核心思想。

**⑤ MMDocRAG（2025）**

当前文档检索增强生成（DocRAG）方法受限于以文本为中心的方式，常常遗漏关键视觉信息；该领域也缺乏用于评估多模态证据选择与整合的鲁棒基准。MMDocRAG 是一个综合性基准，包含 4,055 个专家标注的跨页、跨模态证据链 QA 对。[1](https://mmdocrag.github.io/MMDocRAG/)研究发现，视觉检索器更擅长检索图像内容，文本检索器擅长文本内容，混合检索器能提供更均衡的解决方案。[1](https://mmdocrag.github.io/MMDocRAG/)这一结论直接支持了本项目采用"文本向量检索+视觉原图二次理解"混合架构的设计决策。

**⑥ Multimodal RAG Survey（arXiv 2504.08748, 2025）**

多模态RAG（MRAG）通过将多模态数据（文本、图像、视频）整合进检索与生成过程，增强大型语言模型的能力，克服了纯文本RAG的局限性；MRAG 将多模态检索与生成扩展到框架中，利用来自不同数据类型的上下文信息，从而减少幻觉并增强问答系统的可靠性。[11](https://arxiv.org/abs/2504.08748)近期研究表明，MRAG 在需要同时理解视觉和文本的场景中明显优于传统 RAG。[11](https://arxiv.org/abs/2504.08748)

### 2.3 关键技术对比总结

|维度|纯文本 RAG|本项目多模态 RAG|
|---|---|---|
|文档解析|`page.get_text()` 纯文字提取|VLM 图像解析，输出结构化 JSON|
|检索单元|文本块|文本块 + 表格块 + 图表描述块|
|图表理解|❌ 完全失效|✅ VLM 生成图表自然语言描述|
|生成时上下文|纯文本|检索文本 + 原始页面图像|
|生成模型|Qwen-Turbo（文本LLM）|Qwen2.5-VL（视觉语言模型）|

## 三、系统方案设计

### 3.1 整体架构

系统分为**离线索引**与**在线问答**两条链路，如下图所示：

text

```
【离线索引链路】
用户上传 PDF
    │
    ▼
[PyMuPDF] PDF→页面图片 (page_1.png ... page_N.png)
    │
    ▼
[Qwen2.5-VL API] 逐页图像→结构化JSON提取
    │  {texts:[...], tables:[...], figures:[{fig_id, description},...]}
    ▼
构建 LangChain Document 对象（含元数据: page, type, fig_id）
    │
    ▼
[DashScope text-embedding-v3] 文本向量化
    │
    ▼
[FAISS 本地向量库] 持久化存储

【在线问答链路】
用户输入自然语言问题
    │
    ▼
[FAISS 检索器] top-k=3 相关块
    │  (返回页码、类型、文本内容)
    ▼
读取对应 page_X.png 原图
    │
    ▼
[Qwen2.5-VL API] Prompt(检索文本 + 问题) + 页面原图
    │
    ▼
生成回答（含 [引用：第X页, 图Y/表格Z/文本] 标注）
    │
    ▼
[Gradio 界面] 展示
```

### 3.2 核心模块说明

**模块1：文档解析（parse_doc.py）**

使用 `PyMuPDF` 将 PDF 每页渲染为 PNG 图片（分辨率 2x）。将每张图片以 Base64 编码传入 `Qwen2.5-VL` API，并强制要求输出包含 `texts`、`tables`（Markdown格式）、`figures`（含描述）和 `section_conclusions` 四个字段的 JSON，解析后封装为 LangChain `Document` 对象并附加元数据。

**模块2：向量化入库（build_vectorstore.py）**

使用 `langchain_community.embeddings.DashScopeEmbeddings`（`text-embedding-v3` 模型）将所有 Document 文本内容向量化，通过 `FAISS.from_documents()` 构建索引并持久化到 `faiss_index/` 目录。

**模块3：多模态问答链（rag_chain.py）**

从 FAISS 加载检索器（`k=3`），获取命中块的元数据，读取对应页图像，组装"检索文本 + 原图"多模态 Prompt，调用 `Qwen2.5-VL` 生成带引用标注的最终回答。

**模块4：纯文本基线（baseline_text_rag.py）**

用 `PyMuPDF` 的 `page.get_text()` 提取纯文本，经同样的 Embedding 存入独立 FAISS 索引，调用 `Qwen-Turbo`（纯文本 LLM）生成回答，作为对照组。

**模块5：评测（evaluation.py）**

读取 `test_questions.json` 中的30道测试题，自动调用两套系统并输出回答，人工判分后生成对比表格。

**模块6：界面（app.py）**

基于 `Gradio` 构建 Web 界面，支持文件上传触发解析建库、聊天框交互式问答。

### 3.3 模型选择理由

|组件|选型|理由|
|---|---|---|
|视觉理解模型|Qwen2.5-VL（via DashScope API）|Qwen2.5-VL 已将OCR识别能力提升到新水平，增强了多场景、多语言、多方向的文字识别与定位能力，并在信息提取方面显著增强，满足资质审核和金融业务等领域日益增长的数字化智能需求。[7](https://qwenlm.github.io/blog/qwen2.5-vl/)且无需本地GPU，通过API调用即可。|
|文本嵌入模型|DashScope text-embedding-v3|与VLM同平台，延迟低，支持中英文，免费额度充足|
|向量数据库|FAISS（CPU版）|轻量级，纯本地，无需服务器，`pip install faiss-cpu` 即用|
|纯文本对照LLM|Qwen-Turbo|Qwen2.5-Turbo 和 Qwen2.5-Plus 提供卓越的性价比，同时与 GPT-4o-mini 和 GPT-4o 保持竞争性性能。[12](https://arxiv.org/abs/2412.15115)适合作为对照组快速生成文本回答|
|文档解析|PyMuPDF|无需CUDA，纯CPU即可高速将PDF渲染为图片|

### 3.4 评测指标

|指标|描述|适用场景|
|---|---|---|
|**准确率（Accuracy）**|人工判断回答是否正确（三分制：全对/部分对/错）|30题全部|
|**引用准确率**|回答中引用的页码/图号是否正确|含视觉元素的题目|
|**ANLS**（Average Normalized Levenshtein Similarity）|DocVQA 使用的标准指标：平均归一化编辑相似度[13](https://ukgovernmentbeis.github.io/inspect_evals/evals/multimodal/docvqa/)|需精确答案的题目|
|**拒答率**|系统回答"未找到相关信息"的比例|评估召回能力|
|**图表问题提升率**|多模态RAG相比纯文本RAG在图表题上的准确率提升|核心对比指标|

## 四、初步数据集与实验设想

### 4.1 评测数据集方案

本项目采用**自建小规模精评数据集**与**公开数据集子集**相结合的策略：

**数据集A：公开学术论文 PDF（3份）**

- 选取3篇含有丰富图表（≥5幅图、≥3个表格）的中英文学术论文PDF
- 推荐来源：arXiv.org 计算机视觉/NLP方向近期论文

**数据集B：ChartQA 子集**

- ChartQA 包含 9,600 道人工编写问题及 23,100 道图表摘要生成问题。[5](https://arxiv.org/abs/2203.10244)从中抽取 10 道图表推理题用于评测。

**数据集C：DocVQA 子集**

- DocVQA 是包含 50,000 道问题、覆盖 12,000+ 张文档图像的视觉问答基准。[13](https://ukgovernmentbeis.github.io/inspect_evals/evals/multimodal/docvqa/)从验证集中抽取 10 道结构化文档题（含表格类）。

**自建测试集（test_questions.json）**： 针对数据集A的3份PDF，人工编写30道测试题，分布如下：

|问题类型|数量|示例|
|---|---|---|
|纯文本理解|10题|"第3节的主要结论是什么？"|
|图表理解|10题|"图4中，哪个类别的准确率最高？"|
|表格查询|10题|"表2中，模型A在哪个指标上得分最低？"|

### 4.2 实验设计

**实验一：多模态RAG vs. 纯文本RAG对比**

在相同的30道测试题上，分别运行两套系统，记录回答并人工判分：

|问题类型|纯文本RAG准确率（预期）|多模态RAG准确率（预期）|差距来源分析|
|---|---|---|---|
|图表类（10题）|~10%（无图信息）|~80%|纯文本完全无法理解图表|
|表格类（10题）|~30%（格式错乱）|~75%|PyMuPDF文本提取破坏表格结构|
|文本类（10题）|~75%|~80%|两者相近，差异不大|

具体而言，纯文本 RAG 模型缺乏全面处理多模态信息的能力，导致生成答案的准确性和连贯性较低；在多模态内容一致性得分（MCS）方面，纯文本 RAG 得分极低，因为它无法有效处理视觉内容，限制了将图像或表格纳入回答的能力。[14](https://openreview.net/pdf?id=fMaEbeJGpp)

**实验二：引用定位准确率**

对所有回答检查引用标注（`[引用：第X页, 图Y]`）是否正确，统计引用精确率与召回率。

**实验三：消融实验（可选）**

- 仅用文本描述（无原图）vs. 文本描述+原图，观察原图对回答质量的边际贡献

### 4.3 算力预算

|资源|需求|方案|

|本地计算|CPU-only|普通轻薄本（8GB RAM 即可）|

|Embedding调用|DashScope API|text-embedding-v3 单价极低，30文档全量处理 ≤2元|
|向量存储|FAISS 本地索引|约 50MB 磁盘空间|
|**总计**|**无需GPU**|**约 22 元 API 费用**|

## 五、预期创新点与贡献

1. **零GPU成本的工程落地**：全流程基于 API 调用，在普通 CPU 机器上即可完成，降低多模态RAG系统的实现门槛；
2. **结构化多模态分块**：利用 VLM 对每页文档图像输出结构化 JSON，将"图表描述"作为独立检索单元，弥补传统文本分块对视觉元素的忽视；
3. **图文双轨回答**：检索时定位文本证据，生成时同步传入原始页面图像，实现"语义检索+视觉理解"的协同增效；
4. **系统性对比实验**：在三类问题（文本/图表/表格）上定量对比纯文本RAG与多模态RAG，提供可复现的效果数据。
