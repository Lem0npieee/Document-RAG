## 一、Agentic Engineering 协作原则

### 1.1 每个人独立使用 Claude Code

每位成员在自己的工作目录下使用 Claude Code 进行 AI 辅助开发。关键规则：

- **自己的模块自己负责**，AI 辅助写代码，人做 review
- **提交前互相 review**，AI 生成的代码需要人工验证
- **用 Git Worktree 做并行开发**，避免互相踩脚
- **共享 `CLAUDE.md` 项目规范**，确保 AI 生成代码风格一致

### 1.2 项目级 CLAUDE.md 配置

每位成员在自己负责的子目录放置 `CLAUDE.md`，让 Claude Code 自动理解该模块的上下文：

```
Document-RAG/
├── CLAUDE.md              # 项目级规范（全局共享）
├── src/
│   ├── rag/CLAUDE.md      # 梁力航：问答链路规范
│   ├── parsing/CLAUDE.md  # 梁力航：文档解析规范
│   └── server.py
├── eval/
│   └── code/CLAUDE.md     # 李俊璋：评测脚本规范
├── web/
│   └── CLAUDE.md          # 梁力航：前端规范
└── doc/
    └── tex-report/CLAUDE.md # 丁子睿：LaTeX 报告规范
```

### 阶段 B：自建测试集（第 2-3 天）

#### B1. 文档筛选 + 问题编写 [DATA-04, P0]


| 负责人              | 方式                          |
| ---------------- | --------------------------- |
| **全员**（每人 1-2 篇） | 人工编写 + Claude Code 辅助生成候选问题 |


**分工**：从 KB 已有的 12 篇文档中选 4-5 篇代表性文档：


| 成员  | 负责文档                                                     | 文档特点     |
| --- | -------------------------------------------------------- | -------- |
| 梁力航 | Microsoft 2022 Environmental Sustainability Report (81页) | 最长，结构复杂  |
| 李俊璋 | Boeing 2023 Sustainability Report (100页)                 | 最多页，表格丰富 |
| 丁子睿 | Meta 2023 Sustainability Report (60页) + Coca-Cola (88页)  | 图表丰富     |


**工作流**：

1. 人工浏览分配的 PDF 页面图片（`eval/output/kb/pages/<hash>/`）
2. 用 Claude Code 辅助生成候选问题：

```
# 跟 Claude Code 说：
我已经读了 [文档名] 的第X-Y页。请为这个文档生成8-10道问答对，覆盖以下类型：
- 文本定位：询问特定段落/章节的关键信息
- 表格问答：基于文档中表格数据的查询
- 图表理解：图表趋势、对比、结论
- 跨页延续：需要阅读连续多页才能回答的问题
- 多跳关系：需要综合多个章节/图表才能推理的问题
- 引用追踪：需要明确标出页码和图号的答案

每题格式：
{
  "question": "...",
  "answers": ["标准答案1", "标准答案2"],
  "question_type": "text_location|table_qa|chart_understanding|cross_page|multi_hop|citation_tracking",
  "evidence_pages": [页码],
  "difficulty": "easy|medium|hard"
}
```

1. 人工审核 AI 生成的问题，修改不合理的答案
2. 汇总到 `eval/input/custom/test_questions.json`

## 四、Claude Code 高效使用技巧

### 4.1 模块级 CLAUDE.md

每位成员在自己负责的目录创建 CLAUDE.md，例：

`**eval/code/CLAUDE.md**`（李俊璋）：

```markdown
# 评测模块开发规范
- 所有评测脚本从 eval/code/ 运行
- 数据加载统一用 loader.py 的 load_pdfqa_samples
- 指标计算统一用 metrics.py 的 score_all_metrics
- 输出文件统一放在 eval/output/pdfqa/
- 新增指标先在 metrics.py 实现，再在 run_*.py 调用
- CLI 参数风格参考 run_pdfqa_eval.py
```

### 4.2 常用 Prompt 模板

**写新功能前先问 Claude Code 调研**：

```
我要在 metrics.py 里加 BERTScore 指标。请先看看现有代码结构，告诉我应该怎么加。
```

**写完代码后让 Claude Code 自测**：

```
帮我跑一下 python eval/code/run_pdfqa_eval.py --max-samples 3 --kb-docs-only，
确认新加的指标能正常输出。
```

**写 LaTeX 前提供充分上下文**：

```
基于以下文件帮我写第四章实验：
- eval/output/pdfqa/metrics.json（实验数据）
- eval/output/pdfqa/token_eval_metrics.json（token数据）
- doc/开题报告.md §5（实验设计）
请先生成大纲给我确认，再写完整 LaTeX。
```

### 4.3 注意事项

- **AI 写代码，人做 review**：每段 AI 生成的代码提交前必须人工跑一遍
- **小步提交**：每个小功能完成后立即 commit，方便回溯
- **用 git diff 检查**：提交前 `git diff --staged` 确认改动范围合理
- **不要 trust AI 的评测数字**：评测结果必须自己跑一遍确认
- **LaTeX 编译验证**：AI 写的 LaTeX 提交前必须在本地 `xelatex` 编译通过
